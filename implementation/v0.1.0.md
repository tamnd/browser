# v0.1.0: the shell

Shipped in PR #1.
Goal: a browser you can open, drive with the keyboard, and reshape from the profile folder, even if it is not yet a daily driver.

## Checklist

- [x] SwiftPM executable app, no Xcode project, macOS 14+, hidden title bar
- [x] Workspaces with tab trees in a sidebar, pinned tabs first, depth capped at 4
- [x] Up to 3 vertical panes with split, close, and focus cycling
- [x] Summonable omnibox on cmd+l with scheme, localhost, IP, engine keyword, and dot heuristics
- [x] Command palette on cmd+k over a single command registry, about 30 core commands
- [x] Keymap from keymap.json merged over defaults, null unbinds, live reload
- [x] JSON token themes with light and dark variants, hot reload from themes/
- [x] Page CSS snippets injected from snippets/*.css through one shared user content controller
- [x] JavaScriptCore plugin host with browser.commands, browser.tabs, browser.ui, browser.storage
- [x] SQLite store: history with frecency, single-row session blob, WAL
- [x] Session restore with parent links remapped by index
- [x] Profile scaffold on first launch with README, config, keymap, theme, snippet
- [x] Directory watcher with 300ms debounce for config, themes, snippets
- [x] 32 unit tests, CI on macos-15 with pinned actions

## Notes

- Webviews live in WebViewStore outside SwiftUI, keyed by tab id, so state changes never recreate them.
  The SwiftUI container only reparents when the superview differs.
- Key handling is one NSEvent local monitor so the keymap wins over menu equivalents.
  Overlays consume only escape, arrows, and enter, and let typing fall through to the focused field.
- Tab and Workspace are value types under one @MainActor AppModel.
  That kept mutation code boring and testable, worth keeping.
- WKProcessPool was dropped early, it has been a no-op since macOS 12.
